# 功能开发闭环

## 核心原则

开发功能时，不要直接凭直觉开写。先确认用户目标，再阅读真实代码，找到现有架构中的最小改动路径，然后实现、验证、总结。

默认使用中文思考和回答，除非用户或项目明确要求其他语言。

## 总流程

1. 明确需求
2. 探索代码
3. 制定方案
4. 分步实现
5. 验证功能
6. 审查改动
7. 总结交付

每个阶段都要优先使用能减少猜测的工具。不要问能通过仓库探索回答的问题。

## 阶段 1：明确需求

目标是把用户一句话转成可开发的需求。

需要确认：

- 用户想完成什么
- 功能入口在哪里
- 成功状态是什么
- 失败状态怎么处理
- 是否影响已有功能
- 是否需要兼容旧行为

优先使用：

- 本地代码搜索，而不是立刻提问
- `rg` 查找相似功能、按钮、菜单、命令、服务
- `git status --short` 确认工作区是否已有用户改动

可触发的 Skill：

- `product-design:get-context`：涉及产品界面、流程、交互设计、原型或重设计时使用
- `doc-coauthoring`：需求本身是写规格、方案、PRD 或技术文档时使用
- `openai-docs`：需求涉及 OpenAI / Codex / API 用法时使用

只有在产品取舍无法从代码或上下文判断时才问用户，例如：

- 新功能入口有多个合理位置
- 失败行为有明显产品选择
- 功能范围会显著影响实现复杂度

## 阶段 2：探索代码

目标是理解现有系统，而不是证明自己的第一判断。

常用命令：

```bash
git status --short
git diff --stat
rg --files
rg "关键词"
rg -n "方法名|组件名|状态名|菜单文案"
```

阅读顺序：

1. 入口组件或页面
2. 状态层：ViewModel、Store、State、Reducer
3. 服务层：Service、API、Repository、Platform adapter
4. 数据模型：Model、DTO、Schema、Entity
5. 样式与交互脚本
6. 测试或已有验证命令

优先工具：

- `rg` / `rg --files`：搜索文件和符号
- `sed` / `nl -ba`：读取代码和行号
- `git diff` / `git show`：理解已有修改和历史实现
- `multi_tool_use.parallel`：并行读取多个独立文件

注意：

- 不要用宽泛慢搜索扫整个生成目录；排除 `bin/`、`obj/`、`node_modules/` 等
- 不要修改无关 dirty files
- 如果用户已经改过文件，先理解这些改动，不要覆盖

## 阶段 3：制定方案

目标是让实现路径清楚、范围可控。

方案必须包含：

- 改哪些模块
- 新增哪些状态、方法、组件或服务
- 数据如何流动
- 用户动作如何触发
- 错误和空状态如何表现
- 哪些行为保持不变
- 如何验证

优先选择：

- 复用现有组件和服务
- 跟随项目已有命名和架构
- 小步改动
- 行为级修改优先于大重构
- 性能敏感路径避免同步阻塞

可触发的 Skill：

- `web-design-engineer`：需要构建网页、仪表盘、交互原型、可视化或前端页面时使用
- `frontend-design`：需要明显 UI 设计质量、视觉优化、组件美化时使用
- `Designer`：涉及 logo、视觉系统、品牌界面时使用
- `figma:*`：用户要求写入 Figma、生成 Figma 设计、组件库或图表时使用
- `mcp-builder`：开发 MCP server 或外部服务集成工具时使用
- `Codex-api`：涉及 Codex API、Anthropic SDK 或 Agent SDK 时使用

如果用户明确要求“先给计划”，不要实现；输出决策完整的计划。

## 阶段 4：分步实现

推荐实现顺序：

1. 数据模型或状态
2. 服务接口或业务逻辑
3. ViewModel / Store 方法
4. UI 组件
5. 样式和交互细节
6. 测试或验证入口

优先工具：

- `apply_patch`：手工编辑文件
- `rg`：改完后确认引用和遗漏
- `git diff`：随时查看实际改动
- `multi_tool_use.parallel`：并行读文件，不用于有顺序依赖的编辑

实现约束：

- 不做无关重构
- 不引入不必要抽象
- 不删除用户未要求删除的功能
- 不覆盖用户已有改动
- 不在 UI / 右键菜单 / 渲染路径做同步慢操作
- 不让后台任务假异步：没有真正 `await` 的 `async` 方法可能仍然同步执行

前端功能额外检查：

- 文本不溢出
- 按钮有可理解状态
- 加载、空状态、错误状态合理
- 大列表或多选不会触发大量重复渲染
- 菜单、弹窗、面板可关闭
- 交互不会破坏快捷键、右键、拖拽、焦点和滚动恢复

## 阶段 5：验证功能

根据项目类型选择验证命令。

常见命令：

```bash
git diff --check
dotnet build
npm run build
pnpm build
npm test
pnpm test
```

优先工具和 Skill：

- `webapp-testing`：验证本地 Web 应用功能、截图、浏览器日志
- `playwright-cli`：需要自动点击、填表、截图、端到端交互时使用
- `browser:control-in-app-browser`：需要打开 localhost、file URL 或 Codex 内置浏览器测试时使用
- `Computer Use` / `computer-use:computer-use`：必须操作本机桌面 App、菜单、窗口或原生 UI 时使用

验证时区分：

- 本次新增错误
- 项目已有警告
- 与需求无关的问题

如果构建通过但有警告，要说明警告是否与本次改动相关。

## 阶段 6：审查改动

目标是像代码审查一样找风险，不是只确认“能编译”。

检查清单：

- 新功能是否真正覆盖用户目标
- 是否破坏旧入口
- 状态是否会残留
- 事件订阅是否成对解绑
- 后台任务是否会泄漏或阻塞 UI
- 缓存冷启动时是否有 fallback
- 大数据量是否会退化
- 异常路径是否可恢复
- 删除文件或 API 是否还有引用

优先工具：

```bash
git diff --stat
git diff --check
git diff -- <file>
rg "被删除或改名的符号"
```

可触发的 Skill：

- `frontend-code-review`：审查前端组件、样式、交互和渲染性能
- `ai-code-cleanup`：AI 辅助开发后清理冗余注释、防御性膨胀、无意义抽象和风格漂移

审查输出格式：

- 先列问题，按严重程度排序
- 给文件和行号
- 说明影响
- 给建议修复
- 最后简短说明验证情况

## 阶段 7：总结交付

最终回复要简洁说明：

- 完成了什么
- 改了哪些关键文件
- 验证了什么
- 是否有剩余风险
- 哪些事项按用户要求没有处理

示例：

```text
已完成“批量导出”功能。入口加在文件列表右键菜单中，复用了现有后台任务系统展示进度，导出完成后会刷新目标目录。

验证：
- dotnet build 通过
- git diff --check 通过

注意：
- 构建仍有 3 个既有警告，与本次改动无关。
```

## 常见功能类型与推荐 Skill / 工具

### 新增 UI 页面或组件

使用：

- `web-design-engineer`
- `frontend-design`
- `webapp-testing`
- `browser:control-in-app-browser`

重点：

- 复用现有组件
- 保持视觉一致
- 检查响应式和文本溢出
- 用浏览器或截图验证

### 新增右键菜单、工具栏按钮或快捷键

使用：

- `rg` 查入口和相似菜单项
- `frontend-code-review` 做交互审查
- `webapp-testing` 或 `Computer Use` 验证交互

重点：

- 未选中、单选、多选、全选状态
- 菜单打开速度
- 是否有同步慢操作
- 菜单动作是否使用正确选择集

### 新增后台任务

使用：

- `rg "BackgroundTask|TaskManager|Progress"` 查现有任务系统
- `dotnet build` / 项目构建命令

重点：

- 任务是否真正后台执行
- 是否支持取消
- 完成和失败是否清理
- UI 是否会空面板残留
- 不要共享非线程安全资源

### 新增设置项

使用：

- `rg "Settings|ISettingsService|配置文案"`
- `frontend-code-review`

重点：

- 默认值
- 持久化
- 是否立即生效
- 是否需要重启提示
- 与已有设置 UI 风格一致

### 新增服务或外部 API 调用

使用：

- `openai-docs`：OpenAI 相关
- `mcp-builder`：MCP server 相关
- Web 浏览：只有用户要求最新资料或信息可能变化时使用，优先官方文档

重点：

- 接口抽象
- 错误处理
- 超时和取消
- 缓存策略
- 不阻塞 UI

### 文档、表格、演示、PDF 类功能

按文件类型触发：

- `docx` / `documents:documents`
- `xlsx` / `spreadsheets:Spreadsheets`
- `pptx` / `presentations:Presentations`
- `pdf`

重点：

- 使用对应文件工具链
- 生成后渲染或打开验证
- 不只生成文件，要确认布局和内容正确

## 完成标准

功能完成必须满足：

- 用户目标可达成
- 入口清晰
- 状态正确
- 错误可处理
- 不破坏已有功能
- 构建或测试通过
- 改动范围合理
- 代码风格贴合项目
- 没有明显性能倒退

如果不能完成，要说明：

- 卡在哪里
- 已验证什么
- 缺什么信息
- 下一步怎么做
